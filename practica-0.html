🟨 Práctica 0

Fundamentos, Historia y Estructura de Bases de Datos No Relacionales

Modalidad: Exposición + Podcast + Taller de JSON · Duración: 8 horas (fuera de laboratorio)

🎯 Propósito

Comprender el origen, la evolución y la estructura fundamental de las bases de datos, profundizando en cómo se representa la información en formato JSON antes de trabajar con MongoDB en el laboratorio.

🧭 Contexto conceptual

Antes de usar MongoDB necesitas comprender cómo evolucionaron las bases de datos y por qué JSON se volvió clave para representar información flexible. Esta práctica te da el marco histórico y conceptual para justificar decisiones de modelado documental en el resto del curso.

📚 Parte 1 — Historia y Evolución de las Bases de Datos

Investigar y desarrollar:

  • ¿Qué es una base de datos?
  • ¿Cómo se almacenaba la información antes de las computadoras?
  • Modelos jerárquicos y de red (años 60).
  • Modelo relacional propuesto por Edgar F. Codd (años 70).
  • Aparición y consolidación de SQL.
  • Surgimiento de NoSQL y por qué fue necesario.

Actividad: elaborar una línea del tiempo que incluya:

  • Década
  • Tipo de modelo
  • Características principales
  • Problema que resolvía

🧠 Parte 2 — Introducción a JSON

Antes de trabajar con MongoDB es indispensable comprender cómo se estructura un documento.

Investigar:

  • ¿Qué significa JSON?
  • ¿Qué es un par clave–valor?
  • Tipos de datos en JSON.
  • Diferencia entre objeto y arreglo.

✍️ Parte 3 — Construcción Manual de JSON

En el cuaderno, construir correctamente:

1️⃣ JSON simple

{
  "nombre": "Luis",
  "edad": 18,
  "activo": true
}

Explicar cada elemento.

2️⃣ JSON con arreglo

{
  "materias": ["Programación", "Bases de Datos", "Redes"]
}

3️⃣ JSON anidado

{
  "nombre": "Ana",
  "direccion": {
    "calle": "Reforma",
    "numero": 120,
    "ciudad": "CDMX"
  }
}

4️⃣ Diseño propio: documento "Alumno"

Debe incluir: nombre, edad, grupo, promedio, materias (arreglo), tutor (objeto anidado).

🎙 Parte 4 — Podcast Académico

Tema: ¿Por qué NoSQL cambió la forma de almacenar datos?

  • Breve contexto histórico.
  • Qué es SQL.
  • Qué es NoSQL.
  • Diferencias estructurales.
  • Ejemplo real de aplicación.
  • Conclusión del equipo.

Duración: 5 a 8 minutos · Grabado con celular · Explicación clara y natural.

🎤 Parte 5 — Exposición

Duración: 8 a 10 minutos por equipo. Deben presentar:

  • Línea del tiempo.
  • Ejemplo de JSON construido.
  • Explicación clara de un tipo de base NoSQL.
  • Participación de todos los integrantes.

📦 Entregables

  • Audio del podcast.
  • Evidencia de exposición.
  • Fotografías o PDF del taller de JSON.

🧩 Justificación técnica de los ejercicios

Los ejercicios de línea del tiempo y JSON manual fortalecen la comprensión de cómo y por qué cambió la forma de almacenar datos. Sin este fundamento, el uso de comandos sería mecánico y sin criterio técnico.

🌍 Escenarios reales de aplicación

  • E-commerce: catálogo, carrito y historial de compras.
  • Streaming: perfiles, reproducciones y recomendaciones.
  • Redes sociales: publicaciones, comentarios y reacciones.
  • Sistema escolar: alumnos, materias y evaluaciones.
  • Videojuegos: progreso, logros y sesiones de juego.

🔁 Comparación breve con modelo relacional

En el modelo relacional la estructura se define primero y se mantiene rígida; en el documental, la estructura puede evolucionar según las necesidades del dominio.

⚙️ Rendimiento y escalabilidad

  • Evita escanear colecciones completas cuando un índice puede ayudar.
  • Limita y proyecta campos para reducir transferencia de datos.
  • Cuida el tamaño del documento y el crecimiento de arreglos.
  • Mide consultas críticas y ajusta filtros antes de escalar.
  • Usa paginación o límites en lecturas de alto volumen.

🧪 Qué ocurre internamente en MongoDB

MongoDB almacena documentos como BSON (binario), agrega automáticamente el campo _id y mantiene metadatos para indexación y recuperación. Comprender esto explica por qué el diseño del documento importa desde el inicio.

❗ Errores Comunes

  • Actualizar o eliminar sin filtro específico (o sin _id).
  • Guardar tipos incorrectos (números como texto, fechas sin formato).
  • Diseñar documentos demasiado grandes o con campos duplicados.
  • No documentar la estructura JSON y sus cambios.
  • Confundir colección con documento o arreglo con objeto.

✅ Buenas Prácticas Profesionales

  • Definir convenciones de nombres y mantenerlas en todo el proyecto.
  • Validar datos antes de guardar (tipo, rango, formato).
  • Usar índices en campos consultados con frecuencia.
  • Probar consultas con datos reales y revisar planes de ejecución.
  • Documentar decisiones de modelado y razones de embebido vs referencia.

🧠 Análisis Técnico y Reflexión

  • ¿Qué problema real resolvió NoSQL que el modelo relacional no cubría bien?
  • ¿Qué trade-off aceptas al usar documentos flexibles?
  • ¿Qué riesgos aparecen si el JSON no está estandarizado?
  • ¿Qué decisiones del diseño impactan la escalabilidad futura?

🧭 Retos Aplicados

  • Incluye un JSON con arreglo de objetos y justifica su diseño.
  • Redacta una comparación breve entre normalización y embebido.

🚀 Desafío Nivel Experto

  • Explica qué campos indexarías y por qué.
