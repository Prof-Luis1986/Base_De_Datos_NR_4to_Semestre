🟦 Práctica 1

Base de datos relacional vs base de datos no relacional

🧠 Guía de trabajo

En esta práctica analizarás dos formas distintas de organizar información: las bases de datos relacionales y las no relacionales. El objetivo no es memorizar definiciones, sino comprender cuándo conviene usar cada una y por qué las aplicaciones modernas utilizan cada vez más bases de datos NoSQL basadas en documentos JSON.

Esta práctica es fundamental, ya que todo lo que verás después (JSON, Firebase, Firestore) parte de esta comparación.

🧭 Contexto conceptual

Esta comparación define la base de todas las decisiones de diseño del curso. Entender cuándo una tabla es suficiente y cuándo un documento es mejor te prepara para diseñar soluciones reales, no solo ejercicios.

🎯 Aprendizajes esperados

Al finalizar la práctica, serás capaz de:

  • Explicar qué es una base de datos relacional.
  • Explicar qué es una base de datos no relacional.
  • Identificar diferencias reales entre tablas y documentos.
  • Comprender por qué Firestore utiliza estructuras tipo JSON.
  • Justificar el uso de NoSQL en aplicaciones modernas.

🪑 Actividades en salón

(Trabajo conceptual – sin computadora)

1. Análisis de una base de datos relacional

Observa el siguiente ejemplo de una tabla:

id_alumno nombre grupo edad
1 Ana 3A 17
2 Luis 3B 18

Reflexiona y responde en tu cuaderno:

  • ¿Por qué todos los registros deben tener las mismas columnas?
  • ¿Qué ocurre si un alumno tiene más de un número telefónico?
  • ¿Qué pasa si mañana se necesita guardar redes sociales?
  • ¿La estructura permite crecer fácilmente?

2. Análisis de una base de datos no relacional

Ahora observa el siguiente ejemplo de datos en formato JSON:

{
  "nombre": "Ana",
  "grupo": "3A",
  "edad": 17,
  "telefonos": ["5512345678", "5587654321"],
  "redes": {
    "instagram": "@ana_3a"
  }
}

Reflexiona:

  • ¿Todos los documentos necesitan tener los mismos campos?
  • ¿Qué ventaja tiene que “telefonos” sea una lista?
  • ¿Es obligatorio que todos tengan redes sociales?
  • ¿Qué tan flexible es esta estructura?

3. Comparación guiada

Elabora en tu cuaderno una tabla comparativa como la siguiente y complétala:

Base relacional Base no relacional
Usa tablas Usa documentos
Columnas fijas Estructura flexible
Relaciones con llaves Datos anidados
Cambios complejos Cambios simples
📌 Producto del salón: Tabla comparativa completa + conclusión escrita (mínimo 10 líneas).

🧩 Justificación técnica de los ejercicios

Las actividades comparativas obligan a detectar límites del modelo relacional y fortalezas del documental. Sin esta comparación, cualquier elección tecnológica sería superficial.

🌍 Escenarios reales de aplicación

  • E-commerce: catálogo, carrito y historial de compras.
  • Streaming: perfiles, reproducciones y recomendaciones.
  • Redes sociales: publicaciones, comentarios y reacciones.
  • Sistema escolar: alumnos, materias y evaluaciones.
  • Videojuegos: progreso, logros y sesiones de juego.

🔁 Comparación breve con modelo relacional

El modelo relacional prioriza normalización y consistencia; el documental prioriza flexibilidad y velocidad de lectura con datos embebidos.

⚙️ Rendimiento y escalabilidad

  • Evita escanear colecciones completas cuando un índice puede ayudar.
  • Limita y proyecta campos para reducir transferencia de datos.
  • Cuida el tamaño del documento y el crecimiento de arreglos.
  • Mide consultas críticas y ajusta filtros antes de escalar.
  • Usa paginación o límites en lecturas de alto volumen.

🧪 Qué ocurre internamente en MongoDB

MongoDB lee documentos completos y evita JOINs costosos, pero exige modelado cuidadoso para no duplicar información crítica.

❗ Errores Comunes

  • Actualizar o eliminar sin filtro específico (o sin _id).
  • Guardar tipos incorrectos (números como texto, fechas sin formato).
  • Diseñar documentos demasiado grandes o con campos duplicados.
  • No documentar la estructura JSON y sus cambios.
  • Confundir colección con documento o arreglo con objeto.

✅ Buenas Prácticas Profesionales

  • Definir convenciones de nombres y mantenerlas en todo el proyecto.
  • Validar datos antes de guardar (tipo, rango, formato).
  • Usar índices en campos consultados con frecuencia.
  • Probar consultas con datos reales y revisar planes de ejecución.
  • Documentar decisiones de modelado y razones de embebido vs referencia.

🧠 Análisis Técnico y Reflexión
