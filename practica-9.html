Unidad II · Modelado Documental y Estructura de Datos con JSON
🟦 Práctica 9

Modelado Documental Avanzado y Estructuras Complejas en JSON

Modalidad: Laboratorio · Duración estimada: 2 sesiones

🎯 Propósito

Diseñar y estructurar documentos JSON complejos utilizando objetos anidados y arreglos de objetos, aplicando principios de modelado documental en MongoDB.

🧭 Contexto conceptual

Modelar documentos complejos es lo que diferencia a un usuario básico de un diseñador de bases NoSQL. Aquí aprendes a decidir qué embebes y qué separas.

🧠 Competencia a desarrollar

  • Construye documentos JSON estructurados correctamente.
  • Aplica modelado documental coherente.
  • Justifica decisiones de diseño en bases NoSQL.

🔎 Contexto

En bases de datos relacionales las relaciones se manejan mediante tablas y llaves foráneas. En MongoDB, muchas veces se manejan documentos embebidos y arreglos de objetos, lo que cambia completamente la forma de diseñar. Esta práctica se enfoca en aprender a pensar en documentos.

📚 Parte 1 — Diseño previo (En cuaderno)

Antes de abrir MongoDB, diseñar una base de datos para:

“Plataforma de Streaming”

Debe incluir:

  • Colección: usuarios
  • Colección: peliculas
  • Colección: reproducciones

En el diseño deben decidir:

  • ¿Qué datos se embeben?
  • ¿Qué datos se separan?
  • ¿Por qué?

💻 Parte 2 — Implementación en MongoDB

Paso 1: Crear colección usuarios e insertar un documento complejo:

db.usuarios.insertOne({
  nombre: "Carlos",
  edad: 20,
  suscripcion: {
    tipo: "Premium",
    fecha_inicio: "2024-01-15",
    activa: true
  },
  perfiles: [
    { nombre: "Carlos Jr", edad: 10 },
    { nombre: "Invitado", edad: 25 }
  ],
  historial: [
    {
      titulo: "Interstellar",
      fecha: "2024-02-10",
      minutos_vistos: 120
    },
    {
      titulo: "Matrix",
      fecha: "2024-02-15",
      minutos_vistos: 90
    }
  ]
})

Paso 2: Insertar mínimo 3 usuarios más con estructuras similares pero variando datos.

🔍 Parte 3 — Análisis

Responder:

  • ¿Por qué el historial se guarda como arreglo?
  • ¿Qué pasaría si el historial creciera demasiado?
  • ¿En qué caso convendría separar el historial en otra colección?

📦 Entregables

  • Captura de inserciones.
  • Documento de diseño previo.
  • Respuestas al análisis.

🧩 Justificación técnica de los ejercicios

El diseño previo obliga a justificar embebidos y referencias, clave para rendimiento y consistencia en MongoDB.

🌍 Escenarios reales de aplicación

  • E-commerce: catálogo, carrito y historial de compras.
  • Streaming: perfiles, reproducciones y recomendaciones.
  • Redes sociales: publicaciones, comentarios y reacciones.
  • Sistema escolar: alumnos, materias y evaluaciones.
  • Videojuegos: progreso, logros y sesiones de juego.

🔁 Comparación breve con modelo relacional

En relacional, separar en tablas es norma; en documental, embebido puede reducir JOINs pero aumenta el tamaño del documento.

⚙️ Rendimiento y escalabilidad

  • Evita escanear colecciones completas cuando un índice puede ayudar.
  • Limita y proyecta campos para reducir transferencia de datos.
  • Cuida el tamaño del documento y el crecimiento de arreglos.
  • Mide consultas críticas y ajusta filtros antes de escalar.
  • Usa paginación o límites en lecturas de alto volumen.

🧪 Qué ocurre internamente en MongoDB

MongoDB recupera documentos completos; embebidos se leen en una sola operación, pero arreglos muy grandes pueden degradar el rendimiento.

❗ Errores Comunes

  • Actualizar o eliminar sin filtro específico (o sin _id).
  • Guardar tipos incorrectos (números como texto, fechas sin formato).
  • Diseñar documentos demasiado grandes o con campos duplicados.
  • No documentar la estructura JSON y sus cambios.
  • Confundir colección con documento o arreglo con objeto.

✅ Buenas Prácticas Profesionales

  • Definir convenciones de nombres y mantenerlas en todo el proyecto.
  • Validar datos antes de guardar (tipo, rango, formato).
  • Usar índices en campos consultados con frecuencia.
  • Probar consultas con datos reales y revisar planes de ejecución.
  • Documentar decisiones de modelado y razones de embebido vs referencia.

🧠 Análisis Técnico y Reflexión

  • ¿Cuándo embebido mejora el rendimiento?
  • ¿Cuándo conviene normalizar en colecciones separadas?
  • ¿Cómo afecta el tamaño del documento al rendimiento?
  • ¿Qué decisiones impactan la escalabilidad?

🧭 Retos Aplicados

  • Define límites de crecimiento para el historial y justifícalos.
  • Diseña una versión con historial separado y compara lecturas.
  • Incluye subdocumentos con validación de tipos.

🚀 Desafío Nivel Experto

  • Plantea un esquema alternativo para la plataforma de streaming y defiéndelo.
  • Simula 3 escenarios de consulta y evalúa qué diseño es mejor.
